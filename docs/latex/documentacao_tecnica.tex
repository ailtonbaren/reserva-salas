\section{Arquitetura do Sistema}

\subsection{Visão Geral}
O sistema é uma aplicação web monolítica desenvolvida em Flask, com renderização server-side via Jinja2, persistência em PostgreSQL e interface responsiva com Bootstrap 5.

A escolha por uma arquitetura monolítica é adequada ao escopo acadêmico, pois reduz a complexidade de implantação, facilita a demonstração em ambiente local e mantém as regras de negócio concentradas em uma única aplicação. Mesmo sendo simples, o projeto separa responsabilidades em módulos de rotas, modelos, serviços, templates, arquivos estáticos e testes.

    \centering

\subsection{Stack}
\begin{itemize}
    \item Python 3 e Flask.
    \item Flask-SQLAlchemy e PostgreSQL com psycopg.
    \item Flask-Login para autenticação.
    \item Jinja2 para templates.
    \item Bootstrap 5 por CDN.
    \item pytest para testes automatizados.
\end{itemize}

\subsection{Justificativa da Stack}
\begin{itemize}
    \item Flask foi escolhido por permitir desenvolvimento rápido de aplicações web server-side.
    \item Flask-SQLAlchemy facilita o mapeamento entre classes Python e tabelas do PostgreSQL.
    \item Flask-Login simplifica autenticação, sessão e proteção de rotas.
    \item PostgreSQL foi adotado como banco relacional principal, mais adequado que SQLite para uma proposta próxima de implantação real.
    \item Jinja2 e Bootstrap 5 permitem criar telas responsivas sem necessidade de React, Node.js ou APIs externas.
    \item pytest permite validar regras de negócio de forma automatizada e repetível.
\end{itemize}

\subsection{Camadas}
\begin{itemize}
    \item Aplicação: cria a instância Flask, registra blueprints e configura extensões.
    \item Modelos: define entidades persistidas no banco.
    \item Rotas: organiza os fluxos de autenticação, dashboard, salas, reservas e administração.
    \item Serviços: centraliza regras de negócio de reservas, bloqueios, cancelamentos e configurações.
    \item Templates: renderiza as telas HTML com Jinja2 e Bootstrap.
    \item Testes: valida regras de negócio, rotas e renderização das principais telas.
\end{itemize}

\subsection{Blueprints e Responsabilidades}
\begin{longtable}{|p{4cm}|p{10.5cm}|}
\hline
\textbf{Arquivo} & \textbf{Responsabilidade} \\ \hline
\texttt{auth.py} & Controla login e logout dos usuários. \\ \hline
\texttt{main.py} & Renderiza o dashboard conforme o perfil autenticado. \\ \hline
\texttt{salas.py} & Lista salas, aplica filtros, considera bloqueios do dia atual e exibe horários disponíveis para reserva. \\ \hline
\texttt{reservas.py} & Lista reservas do aluno e processa cancelamentos. \\ \hline
\texttt{admin.py} & Concentra as funcionalidades administrativas: usuários, salas, exclusão controlada de salas, reservas, bloqueios e configurações. \\ \hline
\end{longtable}

\subsection{Fluxo Geral de Requisição}
O fluxo de uma requisição segue a estrutura tradicional de uma aplicação Flask:
\begin{enumerate}
    \item O usuário acessa uma rota pelo navegador.
    \item O Flask direciona a requisição ao blueprint responsável.
    \item A rota valida autenticação e permissões quando necessário.
    \item As regras de negócio são executadas nos serviços.
    \item Os modelos SQLAlchemy consultam ou persistem dados no PostgreSQL.
    \item A resposta é renderizada em template Jinja2 ou redirecionada para outra rota.
\end{enumerate}

\section{Modelo de Dados}

\subsection{Entidades}
\begin{longtable}{|p{4cm}|p{10.5cm}|}

Sala & Representa uma sala de estudo, com nome, localização, capacidade, descrição, status ativo e indicação de bloqueio. \\ \hline

BloqueioSala & Representa bloqueios administrativos de sala por período, com motivo e status ativo. \\ \hline

\subsection{Estratégia de Resposta}
Os riscos de maior impacto estão relacionados ao banco de dados, à integridade das reservas e ao controle de acesso. Para esses pontos, a estratégia adotada foi preventiva: documentação de comandos, validações centralizadas em serviços, testes automatizados e uso de decorators de autenticação e autorização.

Riscos ligados à apresentação e documentação foram tratados com geração de artefatos finais em PDF, arquivo \texttt{comandosFINAL} e arquivo \texttt{validacaoFINAL}.

\section{Segurança e Controle de Acesso}

\subsection{Autenticação}
A autenticação é feita com Flask-Login. Usuários precisam informar e-mail e senha. A senha armazenada no banco não fica em texto puro; ela é transformada em hash por funções do Werkzeug.

\subsection{Autorização}
As rotas administrativas usam o decorator \texttt{admin\_required}. Esse controle impede que alunos acessem funcionalidades de gestão, como cadastro de usuários, cadastro de salas, bloqueios, reservas administrativas e configurações.

\subsection{Proteção de Dados}
O sistema não implementa criptografia avançada, auditoria completa ou LGPD em nível de produção, pois esses pontos estão fora do escopo acadêmico definido. Ainda assim, evita o principal risco imediato ao não armazenar senhas em texto puro.

\section{Testes e Qualidade}

\subsection{Estratégia de Testes}
Os testes automatizados utilizam pytest e o cliente de teste do Flask. A suíte usa banco isolado em memória para executar rapidamente e não depender do PostgreSQL local durante a validação das regras.

\subsection{Cobertura Funcional}
Os testes validam login, erro de login, proteção de rotas, bloqueio de acesso administrativo para aluno, cadastro e edição de usuários, criação de reservas, rejeição de reservas inválidas, cancelamentos, bloqueios, configuração de limite, exclusão controlada de salas, listagem de salas bloqueadas no dia atual e renderização das principais telas.

\section{Implantação Local}

\subsection{Banco de Dados}
O banco principal da aplicação é PostgreSQL. A URL padrão configurada no projeto é:
\begin{verbatim}
postgresql+psycopg://reserva_salas:reserva_salas@localhost:5432/reserva_salas
\end{verbatim}

Caso o ambiente local utilize outro usuário, senha ou banco, a variável \texttt{DATABASE\_URL} deve ser definida antes da execução.

\subsection{Comandos Principais}
\begin{verbatim}
sudo systemctl start postgresql
source .venv/bin/activate
python init_db.py
pytest
python run.py
\end{verbatim}

\subsection{Dados Iniciais}
O script de inicialização cria o administrador, alunos de demonstração, salas iniciais e configuração padrão de limite de reservas ativas por aluno.

\section{Limitações e Evolução}

\subsection{Limitações do Sistema}
\begin{itemize}
    \item Não há envio de notificações por e-mail.
    \item Não há integração com sistema acadêmico real.
    \item Não há cadastro público de usuários.
    \item Não há painel de relatórios estatísticos avançados.
    \item Não há migrações formais com Alembic.
\end{itemize}

\subsection{Possíveis Evoluções}
\begin{itemize}
    \item Adicionar Flask-Migrate para controle formal de mudanças no banco.
    \item Criar relatórios de ocupação por sala e período.
    \item Adicionar paginação nas listas administrativas.
    \item Criar filtros avançados para usuários.
    \item Adicionar notificações por e-mail para reserva e cancelamento.
    \item Criar auditoria administrativa para alterações sensíveis.
\end{itemize}
